Practice 1

Use Meaningful Class Names

Your CSS should describe what, not how

Good names make CSS self-documenting.

Bad names
  • .red-text — tied to appearance
  • .div1 — meaningless
  • .left-bar — tied to position
  • .big — vague
Good names
  • .alert--danger — describes role
  • .card__header — describes structure
  • .sidebar — describes purpose
  • .text-lg — clear utility
💡

Ask yourself: "Will this name make sense in 6 months?"

naming.css 
/* Bad — what happens if it's not red? */
.red-text {
  color: red;
}

/* Good — describes the role */
.alert--danger {
  color: var(--color-danger);
}

/* Bad — tied to layout position */
.left-bar {
  width: 250px;
}

/* Good — describes what it is */
.sidebar {
  inline-size: 250px;
}

/* BEM naming pattern */
.card { /* block */ }
.card__title { /* element */ }
.card__title--highlighted { /* modifier */ }
Practice 2

Keep Specificity Low

Flat selectors are easier to override

High specificity creates a cascade of overrides. Keep it flat.

Avoid
  • Deeply nested selectors
  • IDs for styling
  • !important as a habit
  • Qualifying classes with elements
Prefer
  • Single class selectors
  • BEM-style flat naming
  • @layer for cascade control
  • :where() for zero-specificity resets
specificity.css 
/* Bad — too specific (0-3-1) */
div.page .sidebar .nav-list .item {
  color: blue;
}

/* Good — flat and predictable (0-1-0) */
.nav-item {
  color: blue;
}

/* Bad — hard to override later */
#header .logo {
  font-size: 2rem;
}

/* Good — easy to override */
.header__logo {
  font-size: 2rem;
}

/* Use :where() for zero-specificity */
:where(.reset) a {
  color: inherit; /* easy to override */
}
Practice 3

DRY — Don't Repeat Yourself

Use variables and shared classes

Repetition makes CSS fragile. Centralize shared values.

Strategies:

  • Design tokens via custom properties
  • Shared component classes
  • Utility classes for common patterns
  • Consistent spacing scale
💡

If you change a color in 12 places, you'll miss one. Change it in one variable instead.

dry.css 
/* Bad — repeated values */
.card {
  border-radius: 8px;
  padding: 16px;
  color: #6366f1;
}
.modal {
  border-radius: 8px;
  padding: 16px;
  color: #6366f1;
}

/* Good — centralized tokens */
:root {
  --radius: 0.5rem;
  --spacing: 1rem;
  --color-primary: #6366f1;
}

.card, .modal {
  border-radius: var(--radius);
  padding: var(--spacing);
  color: var(--color-primary);
}
Practice 4

Write Mobile-First CSS

Start small, enhance upward

Base styles for mobile. Media queries add complexity for larger screens.

Desktop-first

Start wide, then max-width to remove features — messy overrides.

Mobile-first

Start simple, then min-width to add features — clean additions.

Benefits:

  • Less CSS overall — base is the simplest layout
  • Media queries only add, never remove
  • Better performance on mobile devices
  • Forces you to prioritize content
mobile-first.css 
/* Base — mobile (no media query) */
.grid {
  display: flex;
  flex-direction: column;
  gap: 1rem;
}

/* Tablet — add columns */
@media (min-width: 640px) {
  .grid {
    flex-direction: row;
    flex-wrap: wrap;
  }

  .grid__item {
    flex: 1 1 calc(50% - 0.5rem);
  }
}

/* Desktop — full grid */
@media (min-width: 1024px) {
  .grid {
    display: grid;
    grid-template-columns: repeat(3, 1fr);
    gap: 1.5rem;
  }
}
Practice 5

Use Shorthand Wisely

Shorthand saves lines but can reset values silently

Shorthand properties are powerful — but use them with intention.

Use shorthand when
  • You're setting all values at once
  • You're starting fresh (resets, base)
  • The values are related
Avoid shorthand when
  • You only want to change one side
  • You'd accidentally reset other values
  • It makes the intent unclear
shorthand.css 
/* Good shorthand — all values set */
.box {
  margin: 1rem 2rem;
  padding: 1rem;
  border: 1px solid gray;
}

/* Careful! background shorthand resets */
.card {
  background-image: url(bg.png);
  background: red; /* resets image! */
}

/* Better — be explicit */
.card {
  background-image: url(bg.png);
  background-color: red; /* no reset */
}

/* Only changing one side? Be specific */
.section {
  margin-block-start: 2rem; /* not margin */
  padding-inline: 1.5rem;   /* left+right */
}
Practice 6

Comment Strategically

Explain why, not what

Comments should add context that the code itself can't express.

Useless comments
  • /* set color */ — obvious
  • /* padding */ — we can see that
  • Commented-out old code
Useful comments
  • Section headers for navigation
  • Explaining browser workarounds
  • Non-obvious magic numbers
comments.css 
/* Bad — says what the code already says */
.title {
  color: red; /* make it red */
}

/* Good — explains WHY */
.title {
  /* Match brand danger color from Figma */
  color: var(--color-danger);
}

/* Good — section headers for scanning */
/* ─── LAYOUT ─────────────────────── */
/* ─── COMPONENTS ─────────────────── */
/* ─── UTILITIES ──────────────────── */

/* Good — explains a workaround */
.modal {
  /* 64px = fixed header height.
     Keeps modal below the header
     on all viewport sizes. */
  top: 64px;
}

/* Good — explains browser fix */
.scroll-container {
  /* Safari needs this for smooth
     scroll inside overflow containers */
  -webkit-overflow-scrolling: touch;
}
Practice 7

Consistent Formatting

Pick a style and stick to it

Consistent code is readable code — automate formatting when possible.

Formatting rules:

  • One property per line
  • Consistent indentation (2 or 4 spaces)
  • Opening brace on the same line as selector
  • Space after colon, before value
  • Semicolon on every declaration
  • Blank line between rule sets

Property order:

1 Layout — display, position, grid
2 Box model — margin, padding, border
3 Typography — font, color, text
4 Visual — background, shadow, opacity
5 Animation — transition, animation
formatting.css 
/* Bad — inconsistent formatting */
.card{
display:flex;
    padding: 1rem;
background: white
}

/* Good — clean and consistent */
.card {
  /* Layout */
  display: flex;
  flex-direction: column;
  gap: var(--space-4);

  /* Box model */
  padding: var(--space-6);
  border-radius: var(--radius-lg);

  /* Typography */
  font-size: var(--text-base);
  color: var(--color-text);

  /* Visual */
  background: var(--color-surface);
  box-shadow: var(--shadow-md);

  /* Animation */
  transition: transform 150ms ease;
}
