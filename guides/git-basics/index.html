Intro

Why Learn Git?

Git is the industry-standard version control system — it tracks every change you make to your code.

Git lets you:

  • Save snapshots of your project at any point
  • Undo mistakes and go back to previous versions
  • Work on features in parallel with branches
  • Collaborate with others without overwriting work

Key concepts:

  • Repository — a project tracked by Git
  • Commit — a saved snapshot of your changes
  • Branch — an independent line of development
  • Remote — a copy of your repo on a server (GitHub)

Basics 1

Setup & Configuration

Install Git and tell it who you are — this info appears in every commit you make.

Install

macOS: comes pre-installed or via Xcode tools

Configure

Set your name and email (once per machine)

Initialize

Create a new repo or clone an existing one

Important:

  • Use the same email as your GitHub account
  • .git/ folder stores all version history
  • .gitignore tells Git which files to skip
setup.sh 
# Check if Git is installed
$ git --version
git version 2.44.0

# Set your identity (once)
$ git config --global user.name "Alice Smith"
$ git config --global user.email "alice@example.com"

# Create a new repository
$ mkdir my-project
$ cd my-project
$ git init
Initialized empty Git repository

# Or clone an existing one
$ git clone https://github.com/user/repo.git

# Check your config
$ git config --list
Basics 2

The Basic Workflow

The core Git loop: make changes, stage them, commit them. Three areas, three steps.

1
Working Directory

Your files as you edit them

2
Staging Area

Changes you've selected for the next commit

3
Repository

The saved history of all commits

Key commands:

  • git status — see what has changed
  • git add — stage changes
  • git commit — save a snapshot
  • git diff — see exact changes
workflow.sh 
# Check the status of your files
$ git status
modified:   index.html
untracked:  styles.css

# Stage specific files
$ git add index.html
$ git add styles.css

# Or stage everything
$ git add .

# Commit with a message
$ git commit -m "feat: add homepage layout"

# See what changed (before staging)
$ git diff

# See what's staged (before committing)
$ git diff --staged

# Unstage a file
$ git restore --staged styles.css
Basics 3

Branches

Branches let you work on features in isolation, then merge them back when ready.

main

The stable, production-ready branch

  • Always working code
  • Where you merge finished features
  • Deploy from here
feature branch

An isolated workspace for new work

  • Safe to experiment
  • One branch per feature/fix
  • Merge or delete when done
branches.sh 
# List all branches (* = current)
$ git branch
* main
  feature/navbar

# Create and switch to a new branch
$ git checkout -b feature/contact-page
# or the newer way:
$ git switch -c feature/contact-page

# Switch between branches
$ git switch main
$ git switch feature/contact-page

# Merge a branch into main
$ git switch main
$ git merge feature/contact-page

# Delete a merged branch
$ git branch -d feature/contact-page
Basics 4

Working with Remotes

Connect your local repo to GitHub (or GitLab) to share code and collaborate with others.

git push

Upload your commits to the remote server

git pull

Download and merge changes from the remote

git fetch

Download changes without merging (safe preview)

Remember:

  • origin is the default name for your remote
  • Always pull before you push
  • Use -u the first time to set upstream tracking
remote.sh 
# Add a remote (after git init)
$ git remote add origin https://github.com/user/repo.git

# See your remotes
$ git remote -v
origin  https://github.com/user/repo.git (push)

# Push for the first time (-u sets tracking)
$ git push -u origin main

# After that, just:
$ git push

# Get latest changes from the team
$ git pull

# Push a feature branch
$ git push -u origin feature/navbar

# Preview remote changes without merging
$ git fetch
$ git log origin/main --oneline
Basics 5

History & Undo

Explore your project's history and safely undo mistakes at any stage.

1
View history

Browse commits with git log

2
Undo changes

Discard edits with git restore

3
Revert commits

Safely undo a commit with git revert

Safety first:

  • Avoid git reset --hard unless you know what you're doing
  • Never rewrite history on shared branches
  • Use revert instead of reset on public branches
history.sh 
# View commit history
$ git log --oneline
a3f8d2c fix: responsive navbar
e7b1a09 feat: add contact page
c4d6f12 feat: add homepage layout

# See what changed in a specific commit
$ git show a3f8d2c

# Discard changes in a file (not yet staged)
$ git restore index.html

# Unstage a file (keep changes)
$ git restore --staged index.html

# Undo the last commit (keep changes)
$ git reset --soft HEAD~1

# Safely revert a specific commit
$ git revert a3f8d2c
# Creates a NEW commit that undoes it

# See who changed each line
$ git blame index.html
Basics 6

Workflow & Best Practices

A clean workflow makes collaboration smooth. Follow these conventions from day one.

Commit messages

Use conventional prefixes: feat, fix, docs, style, refactor

Branch naming

Descriptive names: feature/navbar, fix/login-bug, docs/readme

.gitignore

Exclude node_modules, .env, build files, OS junk

Daily workflow:

  • Pull before you start working
  • Create a branch for each feature or fix
  • Commit small and often with clear messages
  • Push regularly to back up your work
workflow.sh 
# 1. Start your day — get latest changes
$ git pull

# 2. Create a feature branch
$ git switch -c feature/contact-form

# 3. Work, stage, commit (repeat)
$ git add src/contact.html
$ git commit -m "feat: add contact form layout"

$ git add src/contact.css
$ git commit -m "style: add contact form styles"

# 4. Push your branch
$ git push -u origin feature/contact-form

# 5. Merge when ready
$ git switch main
$ git pull
$ git merge feature/contact-form
$ git push
