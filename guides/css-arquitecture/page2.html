Why CSS Organization Matters

Scaling styles without chaos

CSS becomes hard to manage quickly.

  • Leads to conflicts and style overrides
  • Dead code accumulates silently
  • Slows down development over time

Slide 2

Separate Responsibilities

Not all CSS is the same

Each category of CSS has a distinct job.

🌐
Global styles

Foundations — resets, base HTML, tokens

🧩
Component styles

Local UI — scoped, reusable, isolated

⚙️
Feature styles

Business logic — checkout, dashboard, auth

🔧
Utilities

Helpers — spacing, visibility, layout

main.css 
/* ✅ Organized by responsibility */

/* 1. Foundation */
@import './global/reset.css';
@import './global/tokens.css';
@import './global/base.css';

/* 2. Components */
@import './components/button.css';
@import './components/card.css';

/* 3. Features */
@import './features/checkout.css';

/* 4. Utilities */
@import './utils/spacing.css';
Slide 3

Global CSS — Keep It Small

Only for foundations

Global styles are the foundation layer — keep them minimal.

  • Reset / normalize browser defaults
  • Typography — font sizes, line-height
  • CSS variables (design tokens)
  • Base HTML element styles
📁 styles/
📄 reset.css
📄 base.css
📄 tokens.css
📄 globals.css
🎯 Rule: if it's not truly global, it doesn't belong here
reset.css 
/* Minimal, intentional reset */
*, *::before, *::after {
  box-sizing: border-box;
}

body, h1, h2, h3, p, ul {
  margin: 0;
  padding: 0;
}

ul {
  list-style: none;
}

img, video {
  max-width: 100%;
  display: block;
}
base.css 
/* Base HTML elements only */
body {
  font-family: var(--font-sans);
  font-size: var(--text-base);
  color: var(--color-text);
  background: var(--color-bg);
}

a {
  color: var(--color-primary);
  text-decoration: underline;
}
Slide 4

Component Styles

Local, scoped, reusable

Components own their own styles — nothing more.

  • Scoped to the component boundary
  • No side effects on other elements
  • Portable — move the folder, keep the styles
  • Easy to delete without fear
✅ Do

Style only the elements inside your component

❌ Don't

Override parent or sibling components from within

Card.module.css 
/* ✅ Scoped — only Card's internals */
.card {
  padding: var(--space-6);
  border-radius: var(--radius-lg);
  background: var(--color-surface);
  box-shadow: var(--shadow-md);
}

.card__header {
  display: flex;
  align-items: center;
  gap: var(--space-3);
  margin-bottom: var(--space-4);
}

/* ❌ Don't do this in a component */
/* .page-layout { margin: 0 auto } */
Slide 5

Feature Styles

Business logic lives here

Feature styles handle specific page areas or user flows.

  • Checkout flow, dashboard, auth pages
  • Can compose multiple components
  • Usually tied to a route or feature flag
  • Not reused across the app
📁 features/
📁 checkout/
📄 checkout.css
📁 dashboard/
📄 dashboard.css
📁 auth/
📄 auth.css
checkout.css 
/* Feature-specific layout */
.checkout-layout {
  display: grid;
  grid-template-columns: 1fr 380px;
  gap: var(--space-8);
  max-width: 960px;
  margin: 0 auto;
}

.checkout-summary {
  position: sticky;
  top: var(--space-8);
  padding: var(--space-6);
  background: var(--color-surface);
  border-radius: var(--radius-xl);
}

/* Uses Card component internally */
.checkout-summary .card {
  border: none;
}
Slide 6

Utilities — Helpers Only

Small, single-purpose, composable

Utilities do one thing and do it well.

  • Single-purpose classes
  • Spacing, visibility, display
  • Never carry business logic
  • Always the highest layer priority
✅ Good utility

.mt-4 → margin-top: 1rem

❌ Bad utility

.checkout-spacing → too specific

utils/spacing.css 
@layer utilities {
  /* Margin top */
  .mt-0 { margin-top: 0; }
  .mt-2 { margin-top: 0.5rem; }
  .mt-4 { margin-top: 1rem; }
  .mt-8 { margin-top: 2rem; }

  /* Display */
  .flex   { display: flex; }
  .hidden { display: none; }

  /* Visibility */
  .sr-only {
    position: absolute;
    width: 1px;
    clip: rect(0,0,0,0);
  }
}
Slide 7

Putting It All Together

A scalable folder structure

One folder per responsibility — easy to navigate, easy to maintain.

  • New developer can find any style in seconds
  • Each layer is independent and replaceable
  • Works with any framework or bundler
  • Scales from side project to enterprise
🎯 Structure reflects intent
